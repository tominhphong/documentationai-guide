# Design Principles

> Source: `.claude/CLAUDE.md` + README.md

The framework is built on five core principles that shape how agents are organized, how rules are enforced, and how data flows through the system.

## 1. AS-IS Documentation Discipline

**Principle:** Document what the system actually does, not what it could do or what sounds good.

**Implementation:**
- Every agent includes a concrete description of what it outputs and how, not vague aspirations
- Examples show actual command syntax and expected results
- When a tool has limitations, they're stated explicitly
- Knowledge base searches return confidence scores and source attribution
- No "coming soon" or speculative features in agent descriptions

**Why it matters:** Real estate agents need to trust that their AI assistant does exactly what they ask, in exactly the way described. Stale or aspirational documentation breaks that trust immediately.

## 2. Paraphrase, Not Copy

**Principle:** Agents and skills pull insights from knowledge bases but use their own language and examples.

**Implementation:**
- When referencing marketing frameworks (Hormozi, Brunson, Channel Junkies), agents extract principles and apply them to real estate content, not reproduce copyrighted text
- Agents describe their process, not plagiarize examples
- Prompts reference framework concepts ("CON-to-PRO structure," "Daily Seinfeld hook") but build original content
- Blog generator cites frameworks it uses but creates unique structure

**Why it matters:** Original content ranks better, resonates with specific niches (real estate), and avoids copyright issues. The frameworks teach *how to think*, not *what to copy*.

## 3. ClaudeKit-First (MCP-Driven)

**Principle:** All integrations flow through Model Context Protocol (MCP) and Claude Code. No tool access through web UIs or external dashboards.

**Implementation:**
- CRM operations go through `crm_tools/` REST client, not GoHighLevel dashboard
- YouTube ingestion runs via `yt-dlp` script invoked by MCP server, not browser scraping
- Website management goes through Hostinger MCP server or Claude in Chrome, not Hostinger UI
- Gmail/Calendar accessed through Claude.ai integrations (MCP), not third-party tools

**Why it matters:** Single source of truth for all actions. Audit trail is in Claude's session history. No fragmented workflows across multiple UIs. Agent gets full context of what's already been done.

## 4. File Ownership Discipline

**Principle:** Each agent, skill, and rule owns exactly one `.md` file. No shared files, no monolithic instruction documents.

**Implementation:**
- `.claude/agents/youtube-video-pipeline.md` — the video pipeline agent only
- `.claude/agents/email-writer.md` — email generation only
- `.claude/skills/channel-performance.md` — YouTube analytics skill only
- `.claude/rules/content-style.md` — brand voice rules (not agent-specific)
- Each file includes frontmatter: task type, preconditions, output format, success criteria

**Why it matters:** Claude Code loads agents dynamically. When you ask about "video pipeline," it reads exactly one file. When you want to customize email writing, you edit exactly one file. No confusion about which file controls which behavior.

## 5. Frontmatter Discipline

**Principle:** Every agent and skill includes structured metadata in YAML frontmatter. This metadata is machine-readable and human-readable.

**Implementation:**
- Task type: `video-pipeline | email-writing | crm-operation | content-generation | analysis`
- Preconditions: What must be true before this agent runs (e.g., "YouTube channel ingested," "CRM connected")
- Output format: What the agent produces (e.g., "markdown blog post," "JSON guide structure," "CSV contact list")
- Success criteria: How you know it worked (e.g., "Blog includes H2/H3 structure," "Email under 200 words")

Example (conceptual):
```yaml
task_type: content-generation
preconditions:
  - youtube_transcripts_ingested
  - agent_profile_filled
output_format: markdown_blog_post
success_criteria:
  - h2_h3_structure_present
  - under_2500_words
  - includes_video_embed
```

**Why it matters:** When you ask Claude to generate content, it knows exactly what's being asked and can validate its own output before returning it. Metadata also helps Claude pick the right agent from a list.

## 6. Fair Housing Compliance Built-In

**Principle:** Every content piece generated by the framework is Fair Housing-compliant by default.

**Implementation:**
- Brand voice rules explicitly ban phrases like "perfect for families," "dream home," "turnkey"
- Use "buyers" not "families" — avoids steering by family status
- Use "large home" or specify "4-bedroom, 2,500 sq ft" instead of "family home"
- Agents flag any content that risks Fair Housing violations before returning it

**Why it matters:** Real estate marketing has legal requirements. Building compliance into the framework is safer than asking agents to remember rules. Every email, blog post, and video outline is compliant from the start.

## 7. Template Variable Discipline

**Principle:** Personal information (name, phone, market, credentials) is stored once in `config/agent_profile.yaml` and referenced everywhere as template variables.

**Implementation:**
- `{{AGENT_NAME}}` → Auto-fills from config
- `{{AGENT_CITY}}` → Auto-fills from config
- `{{AGENT_CREDENTIAL_LINE}}` → Consistent credibility statement across all content
- Each agent/skill interpolates variables when executing

**Why it matters:** Change your phone number in one place, and every email, blog post, and CRM message gets updated automatically. No outdated contact info in generated content.
